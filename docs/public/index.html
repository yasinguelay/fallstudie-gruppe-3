Dokumentation Fallstudie Kinoticket Reservierungssystem

  1. Anforderung (Manuel)

  1. Ziel

Die Fallstudie Kinoticket Reservierungssystem hilft dabei, dass Kunden eine Sitzplatzreservierung durchführen können. Ein Kino hat in erster Linie dieses System um den Kunden den Reservierungsprozess zu erleichtern, denn der Kunde kann von zu Hause aus über das Internet sich ein Ticket reservieren. Abgesehen davon, hat das Kino den Vorteil, dass die Reservierung von dem Kunden ausgeführt wird und die von Ihm eingegebenen Daten sowie deren Sitzplatz-Präferenzen werden automatisch im System eingetragen und gespeichert.

Abgesehen von der Erleichterung des Prozesses an sich, kann das Kino sich die Ressourcen sparen, indem es theoretisch ohne Personal funktionieren könnte und den gesamten Prozess eines Kinobesuches automatisieren kann.

  1. Motivation

Motivation dieses Systems ist, den Reservierungsprozess eines Kinotickets möglichst unkompliziert und und einfach zu gestalten. Dabei war das Ziel die Benutzeroberfläche minimalistisch wie möglich zu halten damit sich ein Neukunde direkt zurechtfinden kann. Die einzelnen Teilschritte sind unkompliziert, somit ein erstmaliges Benutzen intuitiv und einfach ist.

Der Kunde soll sich direkt zurechtfinden können und ein gewohntes Erlebnis haben während er durch die Seite navigiert.

Für das Kinounternehmen war die Motivation Ressourcen möglichst einzusparen, indem man den Reservierungsprozess mit diesem System automatisiert. Zudem kommt hinzu, dass das Kinoprogramm automatisiert ist und nicht manuell geupdated werden muss, da die Filme von einer Datenbank geschickt werden.

  1. Anwendungsbereiche 

Einerseits kann das System jedes Kino bei sich online implementieren und dabei helfen Sitzplätze und Kinotickets von dem Kunden auswählen und reservieren zu lassen. Das hilft Ressourcen zu sparen, weil es den Reservierungsprozess automatisiert.

Andererseits kann es das Besuchererlebnis verbessern, weil der Kunde selbst entscheiden kann, welchen Platz er reservieren möchte. Das lässt den Besucher der Seite ein Gefühl von mitbestimmung geben.

Zu einem wurde ein Treffen in der gesamten Gruppe angesetzt und ein Meeting  weiteres Treffen wurde explizit für die Gruppenmitglieder ausgelegt, die im Frontend tätig waren.
Bei dem Treffen in der gesamten Gruppe wurden dabei projektspezifische und organisatorische Themen besprochen, die für die Weiterarbeit wichtig waren.

  1. CI/CD

GitHub Actions ist in unserem Fall für die Entwicklung der Continuous Integration zum Einsatz gekommen. Für das Deployment haben wir Heroku GitHub Deploys verwendet.

Dies hatte vor allem drei Gründe. Erstens haben wir schon für die Versionierung mit GitHub gearbeitet und in unserem Team gab es schon viel Vorerfahrung mit dem CI Tool von GitHub, GitHub Actions, in unserem Team. Außerdem ist GitHub sehr gut kompatibel mit der von uns verwendeten Cloud Plattform zum hosten unseres Kinoticket Systems.

Die Umsetzung der Continuous Integration erfolgte dann entsprechend  mit GitHub Actions. Hier definiert man in einer YAML Datei den genauen Ablauf der entsprechenden Tests. In unserem Fall bedeutet dies, dass bei jedem Versuch in den Master Branch zu pushen oder zu mergen, vorher die Tests getriggert werden. Für die Ausführung der Tests wird dann eine Umgebung in Form einer Virtuellen Maschine mit der neuesten Version von Ubuntu Linux generiert. Hier werden dann die nötigen Pakete und Programme installiert, damit der Code auch ausgeführt werden kann. Im letzten Schritt werden dann die Tests ausgeführt. Laufen alle durch, dann geht auch der Push bzw. der Merge durch, falls nicht, dann wird der Push/Merge abgelehnt und die Person, die Pushen/Mergen wollte, muss sich die Testergebnisse anschauen und den Code entsprechend überarbeiten.

Die Umsetzung der Continuous Delivery erfolgte ähnlich wie die zuvor behandelte Continuous Integration. Hier lief es jedoch etwas einfacher ab, da wir keine YAML Datei o.Ä. schreiben mussten. Die Continuous Delivery erfolgte über das in Heroku eingebaute Tool, Heroku GitHub Deploys. Bei diesem Tool wird nach jedem erfolgreichen push oder merge in den Master, d.h. mit erfolgreichem Ablaufen der Tests, die neueste Version des Projekts auf Heroku gezogen. Man hat auch die Möglichkeit ein manuelles Deployment auszuwählen, jedoch haben wir uns dazu entschieden es bei dem Deployment, nach einem erfolgreichen push/merge zu belassen, da hier auch sowohl Qualität als auch aktualität des Codes gewährleistet werden.

  1. Backend:

MongoDB

Wir haben uns für MongoDB als Datenbanksystem entschieden. Der Hauptgrund dafür ist, dass sie weit verbreitet ist und es die am häufigsten verwendete NoSQL-Datenbank ist. Zu dem ist es einfach, diese Datenbank ohne Schema zu installieren, zu verwenden und zu verwenden. Aufgrund der Fähigkeit einer Datenbank ohne Schema definiert der von uns erstellte Code das Schema. Außerdem ist für ein großer Vorteil, dass viele Daten gespeichert werden können, was zu einer schnelleren Abfrageabwicklung beiträgt. Kurz gesagt, mit MongoDB können Sie neue Daten mit der Geschwindigkeit schneller in Ihre Datenbank einfügen insertMany () Funktion. Im Vergleich dazu muss man es bei MySQL einzeln tun.

Node.JS

Als Run-Time-Umgebung haben wir uns für Node.JS entschieden. Wir haben uns schnell darauf geeinigt, da wir den Vorteil darin gesehen haben, dass wir mit dieser Laufzeitumgebung sowohl das Frontend, als auch das Backend der Webanwendungen schreiben können. Wir müssen keine separate serverseitige Programmiersprache verwenden. Zudem haben wir uns auch die  Vorteile von NPM zu Nutze gemacht. NPM verfügt über eine Reihe eigenständiger Tools und Module, die man herunterladen und verwenden kann. Damit kann man die Abhängigkeiten der Projekttools verwalten. Node.js teilt dies mit einem gemeinsamen Repository von NPM, und die Anzahl der darin enthaltenen Module ist erheblich gestiegen. Als letzten Vorteil, der uns wichtig war, war das  Single-Thread-Programm. Als Singlethread-Programm bietet das Tool viele Optionen wie Callback-Methoden und Ereignisschleifen, um mehrere gleichzeitige Clients zu handhaben. Dies kann dazu beitragen, die Wartezeit in den Prozessen des Servers zu beseitigen.

  1. Testing & Qualitätssicherung (Samed)
  1. Mocha

Für das Testing haben wir uns aufgrund den umfangreichen Funktionalitäten und der genauen Dokumentation für Mocha entschieden.

Bei Mocha handelt es sich, dabei um ein sogenanntes Open-Source-Testframework, dass Nutzern die Möglichkeit gibt automatisierte Tests in Browsern und in Node.js durchzuführen.

Mocha definiert sich dabei durch eine klare Struktur bei der Definition von Tests.
Dabei können verschiedene Formen von Testfällen innerhalb der Beschreibungsfunktion (Abfolge von Testfällen), durchgeführt werden.
Asynchrone und synchrone Codes, können unter einem ähnlichen Syntax unkompliziert getestet werden.
Auch die Verwaltung von Tests, gehört zu den Kernfunktionalitäten von Mocha.
Ob die Funktion bzw. der Code den Test bestanden hat oder nicht, wird von Mocha mitgeteilt, indem diese mit “bestanden” oder “nicht bestanden” gekennzeichnet werden.
Jedoch wird für die Überprüfung der Ergebnisse eine Assertion-Bibliothek benötigt, zu diesen gehören Should.js, Expect oder Chai.
Die von uns ausgewählte Assertion-Bibliothek wird in dem folgenden Abschnitt behandelt.

Zusammenfassend konnten wir folgende Vor- und Nachteile erkennen:
Mocha lässt sich sehr leicht bedienen, jedoch ist das Einarbeiten für Anfänger schwieriger, weil das Einrichten von Assertion-Bibliotheken eingerichtet werden mussten, was wiederum fortgeschrittenes Technikwissen erfordert.
Dadurch, dass Mocha sowohl auf dem Browser als auch auf Node.js direkt läuft, war das Testen viel unkomplizierter.


  1. Chai

Bei Chai, wie auch im vorherigen Abschnitt angesprochen, handelt es sich um eine Assertion Library für Browser und Node.Chai kann dabei mit allen Frameworks, die Javascript nutzen kombiniert werden, in unserem Fall mit Mocha.
Darunter gibt es zwei verschieden Interfaces, in dem Chai angeboten wird.Diese unterscheiden sich hauptsächlich darin, dass das erste ein BDD (Behaviour Driven Development) Interface ist und das andere ein TDD (Test Driven Development) Interface.Wir haben uns dabei für das BDD entschieden
BDD ist dabei einfacher und verständlich, da es der englischen Sprache ähnelt und ermöglicht eine lesbaren Stil und Sprache.TDD wird im gegensatz dazu als klassischer Program-Code wahrgenommen.
Dabei bilden drei Assertion Styles die Architektur von Chai, wobei zwei von drei Stilen zu BDD und ein Stil zu TDD gehören.
Die erste von zwei Assertion Styles für BDD, die  Expect Style, erlaubt hierbei die Verkettung von Sprach-Assertion, dafür wird die Anfangs-Funktion “expect” genutzt.Bei dem zweiten sogenannten Should Style, bildet eine Zugriffsfunktion “should” den Anfangspunkt der Sprach Assertions.
Die Auswahl der Assertion library hängt dabei vollkommen von dem Entwickler ab.Da die Kombination von Chai und Mocha am beliebtesten ist, hat sich unsere Gruppe für diese Konstellation entschieden.

Backend-Doku

Swger

Swagger ist eine Sammlung von frei Verfügbaren Werkzeugen, die zum entwerfen, entwickeln und dokumentieren von HTTP-Webservices genutzt werden können.

In unserer Gruppe haben wir uns dazu entschieden Swagger UI zu verwenden, um eine Dokumentation unserer API für das Front End Team zu erstellen. Diese konnte dann vom Front End Team genutzt werden, um strukturiert Daten aus dem Backend auszulesen und auch Daten in das Backend zu schreiben.

Swagger Dokumentiert dabei die einzelnen Schnittstellen, die unsere API bietet und zeigt dann für jede Schnittstelle die entsprechenden Methoden, die ausgeführt werden können. Grundsätzlich arbeitet man dann mit der API in Form von PUT und GET Requests, die aus dem Front End an die API gesendet werden. Diese können dann verschiedene Methoden triggern.

Verdeutlichen lässt sich das am Beispiel der Datenbank der Filme. In diesem Falle bietet die API drei Methoden die ausgeführt werden können. GET, POST und DELETE. Erstere kann dazu verwendet werden die aktuellen Filme und deren Spielzeiten abzurufen, die zweite, um einen neuen Film in die Datenbank zu schreiben und die letzte zum Löschen eines Films. Dabei werden die Daten in Form von JSON Objekten geliefert, die auch in der Datenbank dargestellt sind. Bei PUT Requests sendet das Front End JSON Objekte, die einem gewissen Format entsprechen müssen an die API. Dieses Format ist in der Swagger UI Dokumentation definiert. Unten Abgebildet, der beschriebene Ausschnitt aus der Dokumentation über die Filme.

  1. Seiten, Navigation

Bei der Kino Seite haben wir uns für ein minimalistisches Design entschieden, sodass die Webseite nicht zu überladen ist und den Kunden nicht überwältigt. Es ist auf das Mindeste beschränkt und schlicht gehalten. Die Navigation der einzelnen Teilschritte verläuft auf derselben Seite.

Zunächst sucht man sich einen Film aus, welchen man im Kino anschauen möchte. Sobald man einen Film ausgewählt hat, öffnet sich auf derselben Seite die Vorstellung, in der man Information und Trailer anschauen kann, die Zeiten und Sitzplätze auswählt, sowie Parkettauswahl, als auch die Anzahl der Sitzplätze. Dort wird dann der Gesamtpreis des Tickets zusammengerechnet und dem Kunden angezeigt.

Wenn der Kund fertig ist mit der Auswahl des Filmes und der Sitzplätze, muss man sich zunächst bei der Webseite registrieren, sofern man das nicht schon gemacht hat. Nachdem der Kunde seine Daten angegeben hat, kann der Reservierungsprozess abgeschlossen werden und dem Kunden wird über die im Anmeldungsprozess angegebene E-Mail das Ticket zugeschickt.

  1. Hands-on: Durchführung

Sobald man die Seite besucht, bekommt man das aktuelle Kinoprogramm zu sehen, aus welcher der Kunde auswählen kann welchen Film dieser besuchen möchte.

Klickt man nun auf einen beliebigen Film, öffnet sich unter dem aktuellem Programm

die Vorstellung mit Trailer und einer kurzen Filmbeschreibung.

Nachdem eine Vorstellung ausgewählt wurde, wird der Kunde gebeten die Gesamtzahl der Kinobesucher anzugeben. Zusätzlich hat der Kunde die Möglichkeit zu entscheiden, ob die Sitze sich in Parkett 1 oder 2 befinden. Je nachdem befindet man sich näher oder weiter weg von der Kinoleinwand.

Hat der Kund beispielsweise Tickets für zwei Erwachsene, wird der Preis automatisch zusammengerechnet. Weil der Kunde Parkett 1 (PK1) ausgewählt hat, kann dieser maximal zwei rot-farbige Sitze auswählen und reservieren. Falls der Kunde es sich anders entscheidet, hat er immer die Möglichkeit seine Auswahl komplett rückgängig zu machen.

Ist der Kunde zufrieden mit seiner Sitzauswahl, kann er mit dem nächsten Schritt fortfahren, indem dieser den “Weiter”-Button betätigt, welcher ihn auf einen Login hinweist, wenn dieser noch nicht vollendet wurde, wird man auf eine Login-Seite weitergeleitet.

 

Wenn der Anmeldungsprozess vervollständigt wurde, müssen noch die Restlichen Daten, wie Name, Adresse und Zahlungsmethode (momentan nur Vor Ort Zahlung möglich) ausgefüllt werden.

Hat man letztlich Alles ausgefüllt und auf Buchen geklickt, erhält man ein Pop-up-Fenster, welches die Buchung visuell bestätigt, und den Kunden Informiert, dass das Ticket per Mail zugeschickt wird.

Zusätzlich können weitere Zahlungsmethoden hinzugefügt werden, damit auch online bezahlt werden kann.

Fazit

Zusammenfassend kann gesagt werde, dass die Aufgabenstellung erfolgrecih absolviert wurde. Das Hauptziel war es eine Website zu entwerfen, worauf man ein Ticket für einen Kinobesuch reservieren kann, dies ist uns auch gut gelungen. Unsere Website hilft Kinobetreibern den Prozess des Reservierens zu erleichtern, da der Kunde mit unserer Website von zu Hause aus über das Internet sich ein Ticket reservieren kann. Abgesehen davon, hat unser Kino den Vorteil, dass die Reservierung von dem Kunden ausgeführt wird und die von Ihm eingegebenen Daten sowie deren Sitzplatz-Präferenzen automatisch im System eingetragen und gespeichert.

Abgesehen von der Erleichterung des Prozesses an sich, kann das Kino sich nun die Ressourcen sparen, indem es theoretisch mit weniger Personal funktionieren könnte und den gesamten Prozess eines Kinobesuches automatisieren kann.

Abschließend muss man auch beachten, dass nicht alles perfekt lief. Wie in Kapitel 2 beschrieben, gibt es für uns auch hier Verbesserungsbedarf. Beispielsweise wäre eine offenere Kommunikation in Bezug auf Aufgabenverteilung und technisches Know-How wünschenswert. Dies kann uns in Zukunft einiges an Zeit sparen, sodass wir effizienter arbeiten können, sodass wir auch zusätzliche Features, wie die Erstellung der Benutzeroberfläche für alle Create-Read-Update-Delete (CRUD)-Funktionalitäten hätten einen bauen können.

Trotzdessen konnten wir sowohl unsere kommunikativen, als auch unsere technischen Fähigkeiten stetig verbessern und sind über die Zeit zu einem echten Team geworden, welche alle das gleiche Ziel verfolgt haben.
